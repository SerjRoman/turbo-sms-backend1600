## ERRORS / ОШИБКИ

Ошибки можно создавать свои с более высокой кастомазицией от обычной с использованием **классов** _([смотреть файл с полным примером](src/errors/app.errors.ts))_

### Классы

Для создания классов, используем оператор class и, если хотим, можем его расширить используя extends, по аналогии с интерфейсами

Для определения переменных классов, объявляем их сразу после линии создания класса используя операторы **public** (для переменных, которые должны быть доступны извне класса) и **private** (то же что и публик, только наоборот). После этого можем указать что данное свойство будет доступным только для читания - **readonly** \
**Пример:**

```ts
class AppError extends Error {
  public readonly statusCode: number;
}
```

---

Для инициализации классов, создаем _constructor_ \
Конструктор, хоть и не создается как функция, может принимать аргументы, которые будут использоваться далее \
Если вы делаете наследование другого класса, то **нужно** сделать на начале функции super() и передать в него те параметры, которые примет изначальный класс-родитель. Уже после него, можно начинать делать работу самой инициализации (например, задавать переменные или обработать аргументы)\
**Пример:**

```ts
	constructor(message: string, statusCode: number) {
		super(message);
		this.statusCode = statusCode;
	}
```

---

Для того, чтобы обратиться к свойствам класса, нужно использовать **this** _(Аналогия для Python: self)_
**Пример:**

```ts
this.statusCode = 405;
```

### Обработка ошибок - middlewares

Данные обработчики вызываются когда один из прошлых обработчиков вызвал ошибку. Он размещается на **конце цепочки обработчиков**

Создание обработчика ошибок такое же как и создание обычного middleware _([смотреть файл с полным примером](src/middlewares/error-handler.middleware.ts))_, но как первый аргумент принимает ошибку с стандартным классом _Error_ \
Параметр запроса и параметр ответа также само присутствуют, только не на первом месте

## Prisma

### Adapter

Начиная с Prisma v7, Призма начала требовать adapter для своей работы и, конкретнее, подключения к базе данных

**adapter** _(адаптер)_ - Слой, который позволяет Призме подключится и работать с базами данных. Его можно представить как драйвер у компьютера - также само позволяет системе работать с другой частью корректно.

Как пример адаптера, можно навести **PrismaBetterSqlite3** из _@prisma/adapter-better-sqlite3_ \
После создания, адаптер нужно передать в PrismaClient как параметр \
**Пример _([смотреть исходный файл](src/prisma/client.ts))_:**

```ts
import { PrismaBetterSqlite3 } from "@prisma/adapter-better-sqlite3";
import { PrismaClientC } from "../generated/prisma";

const adapter = new PrismaBetterSqlite3({ url: "../data.db" });

export const PrismaClient = new PrismaClientC({
  adapter,
});
```

### Табличные связи - много подвязок

Если у вас 2 модели связаны между собой более одного раза, то простое использование _@relation(fields: [...], references: [...])_ вызовет ошибку, поскольку Призма начнет путать между собой 2 или более связей.

Для того чтобы исправить данную ошибку, мы должны сказать Призме как связи должны называться при помощи **name**\
name в данном случае обозначает, как Призма должна идентифицировать данную связь, потому **name** должен быть указан на обоих сторонах связи.

**Пример _([User исходный код](prisma/schemes/user.prisma), [Contact исходный код](prisma/schemes/сontact.prisma))_:**

```prisma
    model User {
        id            Int               @id @default(autoincrement())
        contacts      Contact[]         @relation(name: "ContactOwnerToUser")
        contactsOf    Contact[]         @relation(name: "ContactUserToUser")
    }
    model Contact {
        id             Int      @id @default(autoincrement())
        contactUser    User     @relation(fields: [contactUserId], references: [id], name: "ContactUserToUser")
        contactUserId  Int
        contactOwner   User     @relation(fields: [contactOwnerId], references: [id], name: "ContactOwnerToUser")
        contactOwnerId Int
    }
```

## Медиафайлы - multer

### Сохранение в память

Для сохранения файлов в память _(RAM)_, создаем объект multer и передаем в него параметр storage с значением memoryStorage

memoryStorage - объект, который указывает что используем память

**Пример _([смотреть исходный файл](src/middlewares/upload.middleware.ts))_:**

```ts
import multer, { memoryStorage } from "multer";
export const uploadMiddleware = multer({ storage: memoryStorage() });
```

Данный объект multer можно использовать в качестве _middleware_, после которого в параметр запроса будет добавлено свойство file

### _Шакаливание картинок_ - Теория

Картинки, которые будет принимать пользователь от бекенда (например, аватарки других постов) будут занимать время для скачивания (особенно если у пользователя плохой интернет (пример: 3G) или картинки высокого качества (пример: занимают 5МБ пространства)) \
Для этого, мы отправляем пользователю картинку _низкого качества_ перед тем, как отправить полную, чтобы пользователь видел, что он загружает, а не просто кружочек

## Занятие 01.04.2026 - работа с валидацией и аутентификацией.

**При работе с валидацией в yup, yup сверяет данные с заранее описанной «схемой» (правилами).**

_Она проверяет, соответствуют ли данные (например, из формы) типу (строка, число), обязательны ли они, и соответствуют ли формату (email, длина)._

Если данные не подходят, yup возвращает **ошибку**.

---

Для того чтобы указать любо тип данных для схемы, стоит использовать тип **AnySchema**, который импортируется из **yup**.

Типизируем с его помощью схему.

---

```ts
const validated = schema.validate(req.body, {
  abortEarly: false,
  stripUnknown: true,
});
```
---

По умолчанию **abortEarly: true** остановится на первой же
ошибке, поэтому для сбора всех ошибок сразу мы указываем **abortEarly: false**.

По умолчанию **stripUnknown: false** оставляет лишние поля.
Используя stripUnknown: true, мы указываем что стоит удалить всё, чего нет в схеме.

**DTO (Data Transfer Object — объект переноса данных)** — это шаблон проектирования, используемый в программировании для передачи данных между слоями приложения или по сети (между клиентом и сервером). 

**DTO** — это простой объект, содержащий только поля с данными, но **НЕ** содержащий бизнес-логики или методов обработки.
